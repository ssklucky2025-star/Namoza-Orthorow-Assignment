# Task 03 — Integration Design: Landing Page → HubSpot → WhatsApp → Google Ads

**Architecture.** I would not use the native HubSpot embedded form or a no-code connector like Zapier/Make for this. Native embeds hand HubSpot the field validation and styling, which conflicts with the custom-built, PageSpeed-optimised landing page in Task 02, and Zapier/Make introduce polling delays (often 1–5 minutes on standard plans) that put the WhatsApp 2-minute SLA at risk before any message logic even runs. Instead: the form submit triggers a `fetch()` call from the browser straight to a small serverless function (e.g. a Vercel/Cloudflare function or an AWS Lambda behind API Gateway) that we own. That function is the orchestrator and does three things in sequence, not in parallel, because step 2 depends on step 1 succeeding: (1) calls the **HubSpot Contacts API** directly (not the Forms API, which is built for HubSpot-hosted forms and offers less control over the dedupe key) to upsert the contact; (2) on a successful HubSpot write, calls **Karix's WhatsApp Business API** to send the confirmation template message; (3) fires the Google Ads server-side conversion via the **Google Ads API / Enhanced Conversions**, rather than relying solely on the client-side `dataLayer` push, so the conversion is still recorded even if the user closes the tab before GTM's tag fires.

**Biggest failure point.** It's the HubSpot upsert step, specifically deduplication. HubSpot deduplicates contacts on **email by default**, and this form collects no email. Without an explicit fix, every submission creates a new contact even when the same patient calls in twice, which corrupts Lead Status and makes follow-up unreliable. The fix: use phone number as a custom unique-identifier property in HubSpot (a dedicated "WhatsApp Phone" property set as a unique field), and have the orchestrator function search-then-write against that property before creating a contact — search by phone, update if found (appending the new Clinic Preference rather than overwriting it, since a patient asking about two clinics is meaningful data, not a mistake), create if not found. If two patients submit with the same phone but different names, I would not silently overwrite the name; I'd flag the contact with a "Multiple names on this number" internal note and keep the most recent name as primary, since shared family phones are common in this market and a wrong overwrite is worse than a flagged ambiguity a coordinator can resolve on the confirmation call.

**Protecting the 2-minute WhatsApp SLA.** The two realistic breakers are Karix API latency/downtime and the orchestrator function cold-starting under load. Fallback: if the Karix call fails or times out (I'd set an aggressive 15-second timeout), queue the message for retry via a lightweight job queue rather than dropping it, and separately alert on a Slack/email webhook so a human can manually WhatsApp the patient. Monitoring: log a timestamp at form-submit and at WhatsApp-sent, compute the delta on every request, and pipe that into a simple dashboard (or just a scheduled query against the function's logs) with an alert if the rolling p95 exceeds 90 seconds — that gives a warning before the SLA is actually breached, not after.
